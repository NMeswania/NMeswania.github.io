Educational Robot Scheme: Communication

Project Brief

The Educational Robot Scheme was a semester long University project in a group of 6. We were tasked with creating a workshop for students to educate them on coding using the Dyson 360EYE robot. Our solution workshop included 5 challenges for the students to solve using a modified 360EYE: it could have either a gripper-arm or a ping-pong ball launcher. With these attachments, the students would create a script in Google Blockly that would control the robot to solve the challenges.

Other related sub-projects: Electronics Hardware, UI design

Sub-system Brief

I was tasked to find a way to send instructions wirelessly from a tablet/ PC to the robot so students could tell the robot how to solve a challenge. These instructions would be specifically for the new gripper/ ping-pong ball launcher attachments which required some additional electronic hardware and a separate wireless communications channel to the Dyson robot.

Challenges

The communication stream had to be fast enough to process sensor data and move the robot in real-time but also allow the tablet to receive information on the robots’ status.

Communication Method

I decided on using Bluetooth, primarily because the setup was incredibly simple and the communication stream bit-rate was spec’ed to be fast enough. It was also by far the cheapest set of hardware I could find. Now the issue was how to compress the instructions to properly utilise the high speed bit-rate?

Data Compression

With some research, I stumbled across an excellent Arduino-library that encoded commands and data on the MIDI-message format - Standard Firmata. This library would allow me to send instructions from the application in one language to the gripper/ ping-pong ball attachment in Arduino-C. Perfect!

Final Design

The high-speed communication stream allowed for a lot of the data processing to be done by the tablet/ PC.

Further Work

A bit of hardware protection: set boundaries on the speed of which to move the robot arm to stop it moving too fast and end up losing positional accuracy.

Skills

This was a good practice in research and reading/ understanding documentation and code other people have spent years developing.
